为什么现在要区分“项目主线”和“样例验证线”
前面用 `00000422`、TCPPP、decoded 目录做了不少检查。
这些检查有价值，因为它们帮助发现了：
- IMU 轴系语义问题
- 位置观测和速度观测的边界问题
- adapter 层应该怎么抽象

但这些内容本质上是“样例验证线”。
它们的作用是：
1. 验证接口设计有没有明显错误
2. 验证当前原型在一组参考数据上会不会立刻跑崩
3. 帮助暴露框架设计里的薄弱点

它们不等于：
1. 项目主目标已经完成
2. 项目已经进入实用阶段
3. 当前参考数据路线就是最终技术路线

一、项目主线是什么
当前项目主线不是“把某一组数据跑漂亮”，而是先把项目本体搭对。

当前主线应当围绕这几个问题推进：
1. 项目的目标能力是什么
   当前更准确的目标是：
   搭一个可扩展的离线融合研究原型，
   而不是先做成某个具体数据集专用脚本。
2. 项目的正式模块边界是什么
   - `core`
   - `measurements`
   - `adapters`
   - `analysis`
   - `tests`
3. 项目的统一数据契约是什么
   也就是不同来源的数据，最终要转成什么统一观测接口。
4. 配置体系怎么组织
   哪些是通用默认配置，
   哪些是样例验证配置，
   哪些以后会变成场景配置。
5. 未来怎么继续扩传感器
   要先把扩展点设计稳定，
   再去挂新的数据源和新传感器。

二、样例验证线是什么
样例验证线只负责“帮助项目搭建”，不负责定义项目本体。

当前这些内容都属于样例验证线：
1. `00000422` decoded 目录
2. `config_00000422_decoded.json`
3. `运行00000422_推荐配置.bat`
4. `03_results_00000422_decoded_recommended`
5. GREAT-MSF / TCPPP / LCPPP / 外部 `.ins` 的接入检查

这些内容当前的定位应该是：
1. 参考输入样例
2. adapter 设计验证样例
3. IMU / GNSS 语义核对样例
4. 原型结果检查样例

三、当前项目做到哪里了
截至 2026-04-23，当前项目做到的是：
1. 已经不是单文件 demo
2. 已经有明确分层
3. 已经能跑一套离线原型
4. 已经能挂多种 adapter
5. 已经有测试、指标和图表输出

但当前还没有做到的是：
1. 项目路线完全固化
2. 统一观测接口完全定稿
3. 通用多传感器工程化输入全部打通
4. 项目级别的初始化、同步、异常处理全部设计完
5. 真正意义上的“完整项目完成”

所以更准确的表述应该是：
当前项目处于“原型框架搭建阶段”，
而不是“项目已完成，只差接数据”。

四、接下来主线应该做什么
接下来优先级应当回到项目搭建本身：
1. 把正式框架和样例验证内容明确分开
2. 固化项目目标、模块边界和配置分层
3. 明确哪些 adapter 是正式扩展点，哪些只是验证样例
4. 明确统一输入契约
5. 再决定后续传感器扩展和真实数据接入路线

五、你现在看项目时应该怎么理解
1. `config.json`
   看成通用原型默认配置
2. `config_00000422_decoded.json`
   看成样例验证配置，不是项目主配置
3. `00000422` 相关结果
   看成接口和语义检查样例，不是项目最终能力证明
4. 当前代码能跑
   只能说明“原型框架已经能承载一部分流程”
   不能说明“项目已经完整落地”
