项目完成度评估

评估日期：2026-04-27

一、当前项目定位

当前项目应定位为：
离线 ESKF 多传感器融合研究原型。

它当前不是：
1. 实机飞控工程版。
2. 完整 KF-GINS 复现。
3. 已经定型的论文算法。
4. 围绕单一数据集调出来的专用脚本。

当前主线是：
把 IMU/GNSS/baro/mag 的离线 ESKF 项目框架搭完整，
让后续真实日志、鲁棒观测管理、退化场景、baseline 对比和方法迭代有稳定承载。

三、当前项目的强项

1. 项目已经不是单文件 demo。
   目前已经形成较清楚的：
   `core / measurements / adapters / analysis / tests`
   分层。

2. 统一输入契约已经建立。
   这对后续接 PX4、ROS2、Gazebo、VINS、LIO 输出很重要。

3. 观测管理层已经统一。
   不是每个量测各写一套 reject/adaptive 逻辑，
   而是通过 MeasurementManager 统一处理。

4. 观测管理结果已经开始真正进入系统行为。
   当前不仅能记录：
   - reject
   - adaptive R
   - recovery scale
   - reject bypass
   还已经能把这些连续现象接回：
   - quality score
   - state machine
   - metrics
   - fusion_output.csv

5. 初始化不再只是硬编码第一帧。
   当前至少能说明：
   为什么还没初始化，
   用什么 heading，
   是否用了 static alignment，
   是否触发 fallback。

6. 诊断真值和滤波输入已经分层。
   这能降低真值污染风险。

7. 日常检查路径比较清楚。
   当前可以优先跑：
   `python -m unittest discover 05_tests`
   或：
   `04_tools\运行核心检查.bat`

四、当前主要短板

1. 仍是局部 ENU 原型。
   当前不是完整经纬高导航主线，
   也不是完整 KF-GINS。

2. 状态维度仍偏基础。
   当前还没有：
   - 杆臂
   - 外参
   - 时间偏移
   - 气压计 bias
   - 磁航向 bias
   - IMU scale factor

3. 质量评分和状态机已部分进入主线，
   但还没有完全闭合。
   当前已经形成：
   `measurement manager -> quality / mode -> 输出 / metrics`
   的诊断与退化处理链，
   但还没有真正形成：
   `quality / mode -> filter gain / R_eff / gating`
   的正式滤波反馈闭环。

4. 测量模型仍是简化模型。
   GNSS、baro、mag 当前可以支撑原型，
   但还不是高保真真实传感器模型。

5. 时间同步和延迟处理还不系统。
   当前 adapter 有最近邻对齐和容差，
   但没有完整异步队列、重排序、延迟模拟和时间偏移估计。

6. 对比实验还只是模板。
   已经有 fusion_policy 和批处理，
   但还没有系统退化场景和多数据集评估。

7. app.py 仍然偏重。
   已拆出 exporter 和 initialization_controller，
   但 record 构造和主循环诊断仍集中在 app.py。

五、当前阶段是否“走偏”

结论：
前面一度有向实验批处理和代码拆分偏重的趋势，
但当前已经纠回主线。

合理保留的内容：
1. fusion_policy
   因为它提供最基础 ablation 开关。

2. experiment_batch
   因为它是后续对比实验的辅助工具。

3. exporter / initialization_controller 拆分
   因为它降低 app.py 风险。

4. input_quality
   因为它补了真实日志接入前的必要诊断。

需要控制的内容：
1. 不要继续扩大实验系统。
2. 不要继续为了拆而拆。
3. 不要马上追视觉、激光、FGO。
4. 不要把当前状态机包装成完整创新方法。

六、当前完成度判断

如果按“最低完整离线 ESKF 多源融合导航系统”评估：
当前完成度已经进入后半段，
大致可认为在 75% 到 85% 之间。

理由：
1. 基础 ESKF 已能运行。
2. 四类常用观测已接入。
3. 初始化、观测管理、输入契约、结果输出、测试都已有闭环。
4. 多源统计已经能进入 quality、state machine 和 metrics。
5. 已经能承载真实日志 adapter 验证。

如果按“工程可用无人机导航系统”评估：
当前完成度仍然较低。

理由：
1. 没有实时系统。
2. 没有完整传感器误差建模。
3. 没有杆臂、外参、时间延迟。
4. 没有飞控闭环验证。

如果按“论文方法成果”评估：
当前还不能算闭合。

理由：
1. 核心创新点还没有数学化。
2. quality/mode 还没有正式反作用滤波增益或协方差。
3. 缺系统退化场景。
4. 缺严肃 baseline / ablation 结果。

七、下一阶段最应该做什么

下一阶段应继续补项目完成度，
而不是急着扩算法。

建议优先级：

1. 把最低完整主线再收一轮
   继续沿：
   数据读取 -> 初始化 -> predict -> update -> fusion_output.csv -> metrics
   找剩余断口。

2. 继续对齐 measurement manager 和退化处理
   让 reject / adaptive / recover / bypass
   的主线语义更一致。

3. 把 metrics_summary 保持在主线可读水平
   重点服务人工验收，
   不要再扩成报表系统。

4. 等最低完整项目收口后，
   再系统做真实数据验证和实验对比，
   不提前被 batch 细节牵走。

当前项目已经从“能跑的 demo”
推进到了“有分层、有输入契约、有初始化管理、有统一观测管理、有结果诊断、有测试的离线 ESKF 多源融合原型”。

但它还不是成熟工程系统，
