为什么要做这套分层
最初的 demo 写法把传播、更新、输入读取、画图都平铺在 `02_src` 根目录里。
这样上手快，但一旦开始做：
新传感器
创新管理
状态机
真实日志适配
结果诊断
代码很快就会互相缠住。

当前架构
1. core
只放滤波内核。
包括：
- 状态定义
- IMU 传播
- 协方差传播
- 线性更新公共接口

当前 `core` 已经按更正规的方式组织：
- 明确误差状态布局
- 明确过程噪声布局
- 名义状态传播单独成步骤
- 误差状态离散传播单独成步骤
- 局部导航环境单独成模块
- 机械编排单独成模块
- 导航框架自转和速度科里奥利项单独表达
- 二阶离散化和 reset Jacobian 已接入

当前 `core` 的边界是：
- 不关心 csv 列名
- 不直接决定状态机模式
- 不直接画图

2. measurements
每个传感器一个模块。
当前已有：
- gnss_position.py
- gnss_velocity.py
- barometer.py
- mag_yaw.py

这些量测模块已经不只是“转发调用”。
它们负责：
- 量测是否可用
- residual
- H
- R
- 创新统计
- 是否拒绝更新
- 自适应 R 放大

以后要加新传感器，优先在这里新增文件，例如：
- vision_pose.py
- uwb_range.py
- wheel_speed.py

3. adapters
负责把外部数据格式转成统一输入。
当前已有：
- csv_dataset.py
- demo_generator.py

从项目主线角度，现在 `adapters` 最关键的不只是“能读文件”，
而是：
必须把不同来源的数据统一翻译成同一份标准输入契约。
这也是为什么 adapter 可以继续增加，
但 `core` 不应该跟着每种数据格式一起变。

以后你要接：
- PX4 csv
- ROS2 bag
- Gazebo 导出日志
都应该优先在这里做。

4. analysis
负责结果后处理，不污染滤波主干。
当前已经包含：
- 指标计算
- 画图
- 质量评分
- 状态机标签
- 状态机滞回
- 恢复桥接
- 协方差健康分类
- 模式统计
- 原因统计
- 状态机摘要图

这层当前的真实链路大致是：
`sensor freshness + innovations + covariance health`
-> `quality score`
-> `determine_mode`
-> `ModeStateTracker`
-> `CSV / figures / metrics`

5. tests
当前已经不只是最小核心检查。
测试现在覆盖：
- 导航环境相关逻辑
- 健康状态跟踪
- 状态机滞回与恢复
- 指标统计
- 协方差健康分类

长期最关键的约束
1. core 不直接依赖具体日志字段。
2. measurements 不直接管文件读取，但要负责具体量测建模。
3. adapters 不直接写滤波公式。
4. analysis 不反向改滤波状态。
5. tests 要覆盖每次结构化升级，而不只是补一个运行截图。

当前要特别补充的边界
1. `adapters` 里现在既有“正式扩展点”，也有“样例验证入口”。
2. `dx_decoded_flight_csv`、`great_msf_imu_ins`、`dx_imu_external_solution`
   当前更准确的定位是：
   作为项目原型阶段的数据接入验证样例。
3. 这些入口的存在，是为了帮助确认框架该怎么长，
   不是为了提前把项目路线绑定到某一组参考数据。
4. 当前真正应该固化的，是 adapter 的职责边界和统一输出契约，
   不是某个具体样例目录的运行参数。

从项目搭建角度看，当前最该保留的正式内容是
1. 五层目录分工
2. 统一 `SensorFrame / DataFrame` 输入契约
3. 输入契约里的三层字段分级：
   核心输入
   可选观测
   诊断真值
4. `core` 的传播与更新边界
5. `measurements` 的独立建模方式
6. `analysis` 的后处理职责
7. `tests` 的结构化回归机制

当前进一步明确的模块接入规则
1. `core`
   只处理状态、传播、协方差和更新框架。
   不直接接触 `truth_*`。
2. `measurements`
   现在只应该消费观测层视图。
   也就是：
   `gnss_*`
   `baro_h`
   `mag_yaw`
   不应该直接读取 `truth_*`。
3. `analysis`
   当前允许消费诊断/真值层。
   也就是：
   `truth_*`
   这些字段的职责是评估、画图、统计，不是反向驱动滤波。

而当前更应该降级理解为“样例验证内容”的是
1. `00000422` 专用配置
2. 外部 `.ins` 接入检查
3. 某组数据上的推荐参数
4. 某组数据上的推荐 IMU 轴变换

这样做的好处
1. 你后面加传感器时，改动边界清楚。
2. 你后面换数据源时，不需要重写滤波器。
3. 你后面做论文或实验对比时，更容易复现实验链路。
4. 你后面如果继续往 KF-GINS 靠拢，可以继续升级导航主线，而不用再拆 `filter` 主体。
5. 你后面接真实日志时，能先从 `adapters` 入手，不会一上来破坏算法内核。
