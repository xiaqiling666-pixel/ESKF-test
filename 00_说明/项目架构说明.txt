为什么要做这套分层
最初的 demo 写法把传播、更新、输入读取、画图都平铺在 `02_src` 根目录里。
这样上手快，但一旦开始做：
新传感器
创新管理
状态机
真实日志适配
结果诊断
代码很快就会互相缠住。

当前架构
顶层主流程
`app.py` 仍然是离线流水线入口，
但它不应该继续吸收所有业务细节。

当前已经先拆出两类编排辅助：
- `analysis.exporter`
  负责结果导出、图表和指标保存。
- `pipeline.initialization_controller`
  负责初始化阶段判断、等待控制、
  direct / bootstrap 初始化入口选择、
  以及初始化摘要字段更新。

这里的 `pipeline` 不是新的算法层，
它只承接“主流程如何组织已有模块”的职责。
核心初始化模型仍在 `core.initialization`，
量测模型仍在 `measurements`，
结果统计仍在 `analysis`。

1. core
只放滤波内核。
包括：
- 状态定义
- IMU 传播
- 协方差传播
- 线性更新公共接口

当前 `core` 已经按更正规的方式组织：
- 明确误差状态布局
- 明确过程噪声布局
- 名义状态传播单独成步骤
- 误差状态离散传播单独成步骤
- 局部导航环境单独成模块
- 机械编排单独成模块
- 静止粗对准初始化单独成模块
- 时间步长防护单独进入传播策略
- 导航框架自转和速度科里奥利项单独表达
- 二阶离散化和 reset Jacobian 已接入

当前 `core` 的边界是：
- 不关心 csv 列名
- 不直接决定状态机模式
- 不直接画图

当前 `core` 新加入的传播保护职责是：
- 过滤非正 `dt`
- 过滤过小 `dt`
- 过滤或截断过大 `dt`
- 输出传播诊断信息供 analysis 统计

当前 `core` 新加入的初始化职责是：
- 用静止段 IMU 做粗对准
- 给 `roll/pitch` 初值
- 给 `gyro bias` 初值
- 给 `accel bias` 粗初值

当前初始化链路又进一步补了诊断状态：
- WAITING_GNSS
- WAITING_BOOTSTRAP_MOTION
- READY_DIRECT_INIT
- READY_BOOTSTRAP_INIT
- INITIALIZED

并且会记录：
- 当前卡在哪个阶段
- 为什么还没进入初始化
- 当前 heading 来自哪里
- static alignment 是否准备好

当前又进一步补了等待控制：
- 没有 heading 时不会无限等待
- 可以配置最长等待时间
- 可以配置是否允许超时后回退到 `zero yaw` 初始化

这条回退当前默认关闭，
目的是先把控制能力做出来，
而不是默认替你做激进初始化。

这里要明确一个工程事实：
纯静止段对横向 `accel bias` 不完全可观，
所以这层现在给的是“粗估”，
不是最终高可信标定值。

2. measurements
每个传感器一个模块。
当前已有：
- gnss_position.py
- gnss_velocity.py
- barometer.py
- mag_yaw.py

这些量测模块已经不只是“转发调用”。
它们负责：
- 量测是否可用
- residual
- H
- base_R

当前 `measurements` 内部又新增了一层正式的观测管理器：
- measurement manager

这层负责：
- 创新统计
- NIS 门控
- 异常拒绝
- 自适应 R 放大
- 连续拒绝后的恢复阶段渐进重接入
- 按 `fusion_policy` 开关启停上述策略

截至当前版本，这层统一策略已经覆盖：
- gnss_position
- gnss_velocity
- barometer
- mag_yaw

这样做的目的不是多包一层，而是把：
“量测模型怎么写”
和
“鲁棒更新怎么做”
分开。

当前 `fusion_policy` 的定位要说清楚：
- 它是 baseline / ablation 开关
- 它不是新的算法创新本身
- 它让后续可以比较：
  普通更新
  NIS reject
  adaptive R
  recovery scale

当前仍然没有完成的是：
- quality score 直接控制观测权重
- mode state 直接改变量测准入策略
- 退化场景下的统一数学化权重函数

以后如果你要做论文里的：
- quality score
- gating
- mode switch
- adaptive R
- recover

这些更适合继续长在 measurement manager，而不是散落在每个传感器文件里。

以后要加新传感器，优先在这里新增文件，例如：
- vision_pose.py
- uwb_range.py
- wheel_speed.py

3. adapters
负责把外部数据格式转成统一输入。
当前已有：
- csv_dataset.py
- demo_generator.py
- input_quality.py

从项目主线角度，现在 `adapters` 最关键的不只是“能读文件”，
而是：
必须把不同来源的数据统一翻译成同一份标准输入契约。
这也是为什么 adapter 可以继续增加，
但 `core` 不应该跟着每种数据格式一起变。

当前 adapter 统一入口还会生成输入质量报告。
这份报告会统计：
- 核心输入完整性
- 可选观测覆盖率
- 诊断真值覆盖率
- 原始时间戳单调性
- 重复时间戳
- 非正时间步
- 大时间间隔

这些字段会写入 `dataset_source_summary.txt`，
也会以数值形式进入 metrics。
它的定位是数据接入诊断，
不是滤波算法本身。

以后你要接：
- PX4 csv
- ROS2 bag
- Gazebo 导出日志
都应该优先在这里做。

4. analysis
负责结果后处理，不污染滤波主干。
当前已经包含：
- 指标计算
- 结果导出
- 画图
- 质量评分
- 状态机标签
- 状态机滞回
- 恢复桥接
- 协方差健康分类
- 模式统计
- 原因统计
- 状态机摘要图
- 初始化运行级指标
- 初始化可读摘要输出
- 实验批处理指标汇总

这层当前的真实链路大致是：
`sensor freshness + innovations + covariance health`
-> `quality score`
-> `determine_mode`
-> `ModeStateTracker`
-> `CSV / figures / metrics`

当前 `analysis` 对初始化这条线又多承担了一层职责：
- 把 `initialization_summary` 里的文本诊断
  转成可回归的 metrics
- 把初始化阶段、方式、等待时长和 fallback 使用情况
  组织进 `metrics_summary.txt`

这样做的目的不是多写一个 txt，
而是让初始化不再只存在于：
- `dataset_source_summary.txt`
- 裸 metric 键值

而是能直接进入实验结果阅读链路。

当前 `analysis.experiment_batch` 的定位是：
- 顺序运行一组实验配置
- 读取每组结果里的 `metrics.csv`
- 汇总成 `experiment_metrics_summary.csv`
- 额外生成更适合人工快速比较的
  `experiment_metrics_key_summary.csv`

它不改滤波器，
也不定义新算法。
它的作用是补研究证据链里最基础的：
baseline / ablation 对比表。

当前 `analysis.exporter` 的定位是：
- 保存 `fusion_output.csv`
- 保存 `dataset_source_summary.txt`
- 保存 figures
- 保存 metrics

这一步把结果输出职责从 `app.py` 里拆出来，
让 `app.py` 更接近主流程编排，
让 `analysis` 继续承担结果后处理和导出职责。

5. tests
当前已经不只是最小核心检查。
测试现在覆盖：
- 导航环境相关逻辑
- 健康状态跟踪
- 状态机滞回与恢复
- 指标统计
- 协方差健康分类
- 静止粗对准初始化
- 四元数右乘误差注入方向
- 纯旋转 IMU 传播方向

长期最关键的约束
1. core 不直接依赖具体日志字段。
2. measurements 不直接管文件读取，但要负责具体量测建模。
3. adapters 不直接写滤波公式。
4. analysis 不反向改滤波状态。
5. tests 要覆盖每次结构化升级，而不只是补一个运行截图。

当前要特别补充的边界
1. `adapters` 里现在既有“正式扩展点”，也有“样例验证入口”。
2. `dx_decoded_flight_csv`、`great_msf_imu_ins`、`dx_imu_external_solution`
   当前更准确的定位是：
   作为项目原型阶段的数据接入验证样例。
3. 这些入口的存在，是为了帮助确认框架该怎么长，
   不是为了提前把项目路线绑定到某一组参考数据。
4. 当前真正应该固化的，是 adapter 的职责边界和统一输出契约，
   不是某个具体样例目录的运行参数。

从项目搭建角度看，当前最该保留的正式内容是
1. 五层目录分工
2. 统一 `SensorFrame / DataFrame` 输入契约
3. 输入契约里的三层字段分级：
   核心输入
   可选观测
   诊断真值
4. `core` 的传播与更新边界
5. `measurements` 的独立建模方式
6. `analysis` 的后处理职责
7. `tests` 的结构化回归机制

当前进一步明确的模块接入规则
1. `core`
   只处理状态、传播、协方差和更新框架。
   不直接接触 `truth_*`。
2. `measurements`
   现在只应该消费观测层视图。
   也就是：
   `gnss_*`
   `baro_h`
   `mag_yaw`
   不应该直接读取 `truth_*`。
3. `analysis`
   当前允许消费诊断/真值层。
   也就是：
   `truth_*`
   这些字段的职责是评估、画图、统计，不是反向驱动滤波。

而当前更应该降级理解为“样例验证内容”的是
1. `00000422` 专用配置
2. 外部 `.ins` 接入检查
3. 某组数据上的推荐参数
4. 某组数据上的推荐 IMU 轴变换

这样做的好处
1. 你后面加传感器时，改动边界清楚。
2. 你后面换数据源时，不需要重写滤波器。
3. 你后面做论文或实验对比时，更容易复现实验链路。
4. 你后面如果继续往 KF-GINS 靠拢，可以继续升级导航主线，而不用再拆 `filter` 主体。
5. 你后面接真实日志时，能先从 `adapters` 入手，不会一上来破坏算法内核。
