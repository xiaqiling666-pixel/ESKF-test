当前配置分两类

2026-04-28 收口更新
配置系统已经进入主路径校验阶段。
`load_config()` 现在不只是解析 JSON，
还会拦截明显会破坏主链的坏配置。

会被直接拦截的典型错误包括：
1. `gravity` 不是 3 维向量。
2. `gnss_lever_arm_body_m` 不是 3 维向量。
3. process noise、measurement noise、initial covariance 非正或不是有限数值。
4. `time_step_management.max_dt_s <= min_positive_dt_s`。
5. NIS reject threshold 小于 adapt threshold。
6. mode measurement scaling 中存在非正 scale。
7. 当前不支持的导航坐标系。

这层校验的目的不是锁死参数，
而是在真实数据调试前拦住明显错误。

当前新增 GNSS 杆臂配置：
`gnss_lever_arm_body_m`

含义：
GNSS 天线相对 IMU 坐标系原点的固定杆臂，
单位 m。

当前作用：
1. GNSS position update 会补偿天线位置偏移。
2. GNSS velocity update 会补偿角速度引起的天线转动速度。
3. lever arm 诊断会进入 `fusion_output.csv` 和 metrics。

注意：
当前它是固定配置，不是滤波状态。
如果不知道真实安装偏移，先保持 `[0.0, 0.0, 0.0]`。

1. 通用默认配置
`config.json`

用途
- 作为项目原型的默认入口配置
- 用于保留通用结构和默认参数

注意
- 它不应该被某一组样例数据反向绑死
- 后面如果做新的场景配置，不要覆盖它
- 在代码层，它必须声明：
  `config_metadata.profile = default_general`

2. 样例验证配置
例如：
`config_00000422_decoded.json`

用途
- 只用于验证某一组参考数据在当前原型上的表现
- 用于检查 adapter、坐标、IMU 语义、配置组合是否合理

注意
- 它不是项目主配置
- 它不代表项目最终路线
- 它只是一份样例验证 profile
- 在代码层，它不能声明：
  `config_metadata.profile = default_general`

当前代码里的分层约束
1. `01_data\config.json`
   只能使用：
   `default_general`
2. 其他单独配置文件
   只能使用：
   `sample_validation`
   或
   `experiment`
3. 如果角色声明不对，加载配置时会直接报错

当前新增的初始化配置
`initialization`

用途
- 控制静止粗对准是否开启
- 控制静止窗口时长和最小样本数
- 控制多大波动还算“静止”

当前字段
- `static_coarse_alignment_enabled`
- `static_window_duration_s`
- `static_window_min_samples`
- `static_max_accel_std_mps2`
- `static_max_gyro_std_radps`
- `static_gravity_norm_tolerance_mps2`
- `bootstrap_min_dt_s`
- `bootstrap_min_horizontal_displacement_m`
- `heading_wait_timeout_s`
- `zero_yaw_fallback_enabled`

说明
- 这组配置主要影响：
  起飞前静止段
  roll/pitch 初值
  gyro bias 初值
  accel bias 粗初值
- 纯静止段下，横向 `accel bias`
  只能做粗估，
  不要把它当成完整标定结果。
- `heading_wait_timeout_s`
  控制在没有 heading 来源时最多等多久。
- 如果 `zero_yaw_fallback_enabled = true`
  并且超过等待时间，
  系统可以退化为：
  用 `yaw = 0`
  做 direct init。
- 当前默认是关闭的，
  也就是只提供这条回退能力，
  不默认启用。

当前初始化诊断输出
当前主流程会在初始化摘要里记录：
- `initialization_phase`
- `initialization_reason`
- `initialization_ready_mode`
- `heading_source`
- `static_alignment_ready`
- `static_alignment_reason`

当前这些字段不再只停留在：
`dataset_source_summary.txt`
里。
它们现在还会以两种形式进入分析输出：
1. 作为运行级 metrics
   例如：
   `initialization_mode_direct_flag`
   `initialization_mode_bootstrap_position_pair_flag`
   `initialization_static_coarse_alignment_used_flag`
   `initialization_zero_yaw_fallback_used_flag`
   `initialization_wait_s`
2. 作为 `metrics_summary.txt`
   顶部的人类可读初始化摘要

当前阶段含义
- `WAITING_GNSS`
- `WAITING_BOOTSTRAP_MOTION`
- `READY_DIRECT_INIT`
- `READY_BOOTSTRAP_INIT`
- `INITIALIZED`

这层的目的不是多打印几行字，
而是把“为什么现在还没初始化”变成可解释状态。

当前新增的时间步长防护配置
`time_step_management`

用途
- 约束滤波传播允许使用的 `dt`
- 防止日志时间戳异常直接把传播炸坏
- 为后面真实日志接入提供统一 `dt` 策略

当前字段
- `min_positive_dt_s`
- `max_dt_s`
- `skip_large_dt`

说明
- `dt <= 0`
  当前直接跳过传播
- `dt` 小于最小正步长
  当前记为异常并跳过
- `dt` 大于最大步长
  当前默认跳过
  如果后面把 `skip_large_dt = false`
  则改为截断到 `max_dt_s`

当前创新管理配置补充
`innovation_management`

当前已统一覆盖：
- `gnss_pos`
- `gnss_vel`
- `baro`
- `mag_yaw`

说明
- 现在不是只有 GNSS 才有统一 `NIS / reject / adapt`
- `baro` 和 `mag_yaw` 也已经接入 measurement manager 的统一 policy
- 这让观测管理层在当前项目里第一次真正覆盖了所有现有观测类型

当前融合策略开关
`fusion_policy`

用途
- 支持 baseline / ablation
- 不改量测模型本身
- 只控制 measurement manager 是否启用对应鲁棒策略

当前字段
- `use_nis_rejection`
- `use_adaptive_r`
- `use_recovery_scale`

说明
- 三个字段默认都是 `true`
- 默认行为保持当前完整观测管理策略
- 如果要做最基础 baseline，
  可以在单独实验配置里关闭这些开关
- 当前这只是实验开关，
  还不是“质量驱动融合方法”本身
- `quality_score` 和状态机目前仍主要用于诊断与统计，
  尚未直接反作用于观测权重

当前实验配置模板
以下配置用于 baseline / ablation，
默认都使用 `standard_csv` demo 数据，
目的是先固定实验开关和结果目录，
不是绑定某一组真实样例。

1. `config_experiment_baseline_eskf.json`
   - 关闭 NIS rejection
   - 关闭 adaptive R
   - 关闭 recovery scale

2. `config_experiment_nis_reject.json`
   - 开启 NIS rejection
   - 关闭 adaptive R
   - 关闭 recovery scale

3. `config_experiment_adaptive_r.json`
   - 开启 NIS rejection
   - 开启 adaptive R
   - 关闭 recovery scale

4. `config_experiment_adaptive_r_recovery.json`
   - 开启 NIS rejection
   - 开启 adaptive R
   - 开启 recovery scale

5. `config_experiment_full_method.json`
   - 当前完整原型策略模板
   - 目前和完整 measurement manager 策略一致
   - 注意：
     quality score 和状态机仍主要用于诊断，
     尚未直接反作用于滤波权重

当前实验批处理入口
`python 02_src\run_experiment_batch.py`

用途
- 顺序运行当前实验配置模板
- 读取每个结果目录下的 `metrics.csv`
- 汇总为：
  `03_results_experiment_batch\experiment_metrics_summary.csv`
- 同时生成紧凑版：
  `03_results_experiment_batch\experiment_metrics_key_summary.csv`

说明
- 这个入口会生成结果目录
- 日常开发检查仍优先使用：
  `python -m unittest discover -s 05_tests`
- 做实验对比时再运行批处理入口

后面建议
1. 通用配置继续保留在 `01_data` 根目录
2. 样例验证配置可以继续按数据编号逐步增加
3. 等项目配置体系进一步稳定后，再决定是否单独拆成：
   `通用配置`
   `样例配置`
   `实验配置`
