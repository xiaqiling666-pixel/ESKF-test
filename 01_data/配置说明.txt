当前配置分两类

1. 通用默认配置
`config.json`

用途
- 作为项目原型的默认入口配置
- 用于保留通用结构和默认参数

注意
- 它不应该被某一组样例数据反向绑死
- 后面如果做新的场景配置，不要覆盖它
- 在代码层，它必须声明：
  `config_metadata.profile = default_general`

2. 样例验证配置
例如：
`config_00000422_decoded.json`

用途
- 只用于验证某一组参考数据在当前原型上的表现
- 用于检查 adapter、坐标、IMU 语义、配置组合是否合理

注意
- 它不是项目主配置
- 它不代表项目最终路线
- 它只是一份样例验证 profile
- 在代码层，它不能声明：
  `config_metadata.profile = default_general`

当前代码里的分层约束
1. `01_data\config.json`
   只能使用：
   `default_general`
2. 其他单独配置文件
   只能使用：
   `sample_validation`
   或
   `experiment`
3. 如果角色声明不对，加载配置时会直接报错

当前新增的初始化配置
`initialization`

用途
- 控制静止粗对准是否开启
- 控制静止窗口时长和最小样本数
- 控制多大波动还算“静止”

当前字段
- `static_coarse_alignment_enabled`
- `static_window_duration_s`
- `static_window_min_samples`
- `static_max_accel_std_mps2`
- `static_max_gyro_std_radps`
- `static_gravity_norm_tolerance_mps2`

说明
- 这组配置主要影响：
  起飞前静止段
  roll/pitch 初值
  gyro bias 初值
  accel bias 粗初值
- 纯静止段下，横向 `accel bias`
  只能做粗估，
  不要把它当成完整标定结果。

当前新增的时间步长防护配置
`time_step_management`

用途
- 约束滤波传播允许使用的 `dt`
- 防止日志时间戳异常直接把传播炸坏
- 为后面真实日志接入提供统一 `dt` 策略

当前字段
- `min_positive_dt_s`
- `max_dt_s`
- `skip_large_dt`

说明
- `dt <= 0`
  当前直接跳过传播
- `dt` 小于最小正步长
  当前记为异常并跳过
- `dt` 大于最大步长
  当前默认跳过
  如果后面把 `skip_large_dt = false`
  则改为截断到 `max_dt_s`

后面建议
1. 通用配置继续保留在 `01_data` 根目录
2. 样例验证配置可以继续按数据编号逐步增加
3. 等项目配置体系进一步稳定后，再决定是否单独拆成：
   `通用配置`
   `样例配置`
   `实验配置`
